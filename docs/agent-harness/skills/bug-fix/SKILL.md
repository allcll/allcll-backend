# 올클 버그 수정 가이드 (v2 - iteration-1 측정 반영)

이 skill은 **fix 영역 baseline 관찰**(2026-04-25) 과 **iteration-1 측정**(2026-05-11, 4 evals × with/without) 결과를 근거로 작성됨.

## 작업 범위 정의

### 적용 대상

- 보고된 버그 수정
- 검증 로직 추가 (validation, 권한 등)
- 응답 형식 보완
- 작은 기능 추가 (단일 엔드포인트 수준)

### 적용 안 함

- 큰 기능 추가 (여러 컴포넌트 신설) — 별도 설계 단계 필요
- 도메인 모델 변경 — 영향 범위 너무 넓음
- 동작 보존 작업 — refactoring skill 사용

근거 (iteration-1 eval-2): "Optional 처리를 더 안전하게 개선해줘" 같은 요청은 동작 보존이 목적이면 refactoring 영역. with_skill 은 decision.md 로 분기 보고, without_skill 은 그냥 진행.

## 작업 순서 (의무)

### Step 1: 버그 재현 또는 시나리오 명확화

수정 시작 전:

1. 기존 코드의 **현재 동작** 파악
2. **기대 동작** 명시
3. 둘의 차이 = 수정해야 할 것

가능하면 **재현 테스트 먼저 작성**:

- 빨간색 (실패) 테스트 = 버그의 정의
- 수정 후 초록색 (통과) = 버그 해결 증거

baseline 관찰 (2026-04-25): 이 단계 통째로 빠뜨림.
"테스트는 변경 후에도 안 짬" → 회귀 방지 없음.

### Step 2: 기존 패턴 탐색

새로 만들지 말고 **기존 패턴 답습**:

- 비슷한 작업이 프로젝트에 있는지 Grep
- 적용 라이브러리·프레임워크 확인
- 컨벤션 발견하면 그대로 따름

baseline 관찰: 이 단계는 잘 됨 (AdminNoticeApi 패턴 발견). 강화 유지.

### Step 3: 영향 범위 점검

수정 위치 외에 같은 문제가 있을 수 있는 곳:

- 같은 도메인의 다른 API
- 같은 패턴의 다른 클래스
- 호출자 (이 코드를 쓰는 곳)

#### 전수 grep 의무

"비슷한 X 도 있을 수 있겠지" 가 아니라 **실제 grep 으로 전수 확인**:

- DTO 검증 추가 시 → `find . -name "*Update*Request.java" -o -name "*Create*Request.java"` 같이 같은 패턴 파일 전체 grep
- 컨트롤러 결함 발견 시 → 같은 종류 컨트롤러 (`Admin*Api.java` 등) 전체 grep
- 결과는 notes 에 박제 ("X개 파일 점검, Y개 동일 결함 발견")

근거 (iteration-1 eval-3): with_skill 은 6개 `*Update*Request` 전수 점검으로 비대칭이 `UpdateNoticeRequest` 하나뿐임을 확인. without_skill 은 "다른 비슷한 곳도 있을 수 있다" 정도로 끝냄.

#### 컨테이너 어노테이션 누락 점검 (필수)

검증·보안 어노테이션을 추가·수정할 때, 그 어노테이션이 동작하려면 **컨테이너에 트리거 어노테이션이 있어야 한다**. 둘이 짝이라 한쪽만 손대면 무력.

| 추가 대상 | 함께 점검할 컨테이너 |
|---|---|
| DTO 필드의 `@NotNull` / `@NotBlank` / `@Size` 등 | 컨트롤러의 `@RequestBody` / `@ModelAttribute` 가 **`@Valid`** 를 가졌는가 |
| 메서드 파라미터의 `@NotBlank` / `@Min` / `@Pattern` 등 | 컨트롤러 클래스에 **`@Validated`** 가 있는가 |
| `@PreAuthorize` / `@Secured` | **메서드 시큐리티 활성화** (`@EnableMethodSecurity`) 여부 |
| `@Cacheable` / `@CacheEvict` | **캐시 활성화** (`@EnableCaching`) 여부 |
| `@Async` / `@Scheduled` | **각 기능 활성화** (`@EnableAsync` / `@EnableScheduling`) 여부 |

근거 (iteration-1 eval-1): `LoginRequest` 에 `@NotBlank` 만 추가하고 `AuthApi#login` 의 `@Valid` 누락을 놓치면 검증이 트리거 안 됨 → 빈 값 통과. with_skill 은 분리 보고로 발견, without_skill 은 추가만 하고 끝냄.

작업 절차:

1. 작업 대상 어노테이션의 트리거 컨테이너를 위 표에서 식별
2. 컨테이너 위치 grep — 추측 금지
3. 트리거 어노테이션 부재 시 → **분리 보고 항목으로 즉시 박제** (사용자 결정 필요)

#### 발견 시 처리

- 함께 수정 가능 → 범위 명시 후 수정
- 별도 작업 가치 → 사용자에게 분리 보고

### Step 4: 구현

기존 패턴 따라 최소 변경:

- 새 의존성 추가 최소화
- 기존 컨벤션 유지
- 변경 라인 수 작게

### Step 5: 검증 (의무)

❌ **컴파일만으로 작업 종료 금지**

다음을 모두 수행:

1. **컴파일**: `./gradlew compileJava -q`
2. **관련 테스트**: `./gradlew :test --tests "*관련패턴*"`
3. **재현 테스트 추가** (Step 1에서 못했으면 여기서):
    - 버그 케이스 → 통과 확인
    - 정상 케이스 → 통과 확인
4. **전체 테스트** (영향 범위 큰 변경의 경우): `./gradlew :test`

baseline 관찰 (2026-04-25): 컴파일만 함. 테스트 단계 누락.
**이게 가장 큰 약점**이라 의무화 강조.

## 작업 범위 분리 원칙

작업 중 발견한 인접 문제는 **명시적으로 분리**:

> "원래 보고 받은 X는 수정했습니다.
> 작업 중 Y 문제도 발견했는데, 이건 [이유]로 별도 판단이 필요해
> 이번엔 손대지 않았습니다. 추가 작업 원하시면 알려주세요."

baseline 관찰: 이미 잘 됨 (HttpMessageNotReadable 처리 분리 보고).
강화 유지.

## 자가 보고 형식 (필수)

작업 끝나면 다음 5개 섹션을 **모두 명시**. 해당 사항 없으면 "해당 없음" 으로 박제 — 누락은 작업 미완료로 간주.

1. **변경 파일 목록 + 라인 수**
2. **변경 핵심 한 줄 요약**
3. **검증 단계 결과** (컴파일 / 테스트 / 추가 작업)
4. **분리한 인접 문제** (있으면 항목별로 / 없으면 "해당 없음")
5. **다음 단계 제안** (있으면 / 없으면 "해당 없음")

근거 (iteration-1): 스킬 미적용 baseline 은 4·5 항목을 자주 빠뜨림. 사용자가 인접 문제·다음 단계를 직접 물어야 알게 되는 비효율 발생. **빈 항목도 명시하면 "점검 안 함" 과 "점검 후 없음" 이 구별된다** — 이 구별이 사용자 신뢰의 핵심.

## 절대 원칙

- 컴파일만 통과시키고 작업 완료 선언 금지
- 기존 테스트가 깨지면 무시하지 말 것 (분석 → 정당화 또는 롤백)
- 버그 재현 테스트 없이 "고쳤다"고 하지 말 것
- 사용자가 명시한 범위 외 임의 확장 금지
- 응답 형식·API 명세 변경 시 사용자 확인 필수
- **검증 어노테이션은 컨테이너 트리거와 짝** — 한쪽만 손대지 말 것 (Step 3 표 참고)
