# allcll 리팩토링 가이드

## 범위 프레임

작업 시작 전에 **허용·금지 범위**를 명시적으로 정의한다. 애매하면 사용자에게 물어본다.

**허용 (예시)**:

- 지정된 클래스·메서드 내부 로직 수정
- 필요 시 최소한의 헬퍼 메서드 추가

**금지 (기본값)**:

- 다른 메서드 시그니처 변경
- 테스트 코드 수정 (수정 없이 통과해야 함)
- 기존 주석 제거·수정
- 네이밍/스타일 임의 변경
- 의도하지 않은 부가 리팩토링 (새 인터페이스, 불필요한 추상화)

## 작업 순서

1. **Baseline 확보**: 수정 전 관련 테스트가 통과하는지 먼저 확인
    - 멀티 모듈 프로젝트임: `./gradlew :test --tests "*패턴*"` (접두 `:test` 주의)
2. **리스크 분석**: thread safety, 외부 제약(rate limit 등), 기존 TODO 주석 확인
3. **안전 확신 후 구현**. 불확실하면 먼저 보고
4. **재검증**: 같은 테스트 다시 실행. 수정 없이 통과 확인
5. **자가 보고**: 변경 파일 목록, 라인 수, 테스트 결과, 적용 안 한 개선점

## 실패 처리 원칙

테스트 실패 시:

- ❌ **테스트를 수정해서 통과시키지 않는다**
- ✅ 실패 원인을 분석. 구현 문제면 롤백 후 재시도
- ✅ 기존 라이브러리/프레임워크의 숨겨진 동작(예: Spring 내부 상태)을 발견할 기회로 삼는다

## 주의할 외부 제약

- **외부 API 연동 코드 (SJPT 서버 등)**: `Thread.sleep`이나 rate limit 관련 코드는
  의도적일 수 있음. 제거 전 운영자 확인 필수. (근거: AdminBasketService)
- **스케줄러 pool size, fixedRate/fixedDelay**: 설정 변경 시 전체 주기에 영향
