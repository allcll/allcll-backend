# allcll 리팩토링 가이드 (v2 - iteration-1 측정 반영)

이 skill 은 **Session 4-C 경험** + **iteration-1 측정** (2026-05-11, 4 evals × with/without) 결과를 근거로 작성됨.

## 범위 프레임

작업 시작 전에 **허용·금지 범위**를 명시적으로 정의한다. 애매하면 사용자에게 물어본다.

**허용 (예시)**:

- 지정된 클래스·메서드 내부 로직 수정
- 필요 시 최소한의 헬퍼 메서드 추가

**금지 (기본값)**:

- 다른 메서드 시그니처 변경
- 테스트 코드 수정 (수정 없이 통과해야 함)
- 기존 주석 제거·수정
- 네이밍/스타일 임의 변경
- 의도하지 않은 부가 리팩토링 (새 인터페이스, 불필요한 추상화)

CLAUDE.md 의 '변경 범위 가이드' (파일 3개·50줄·함수 2개) 한도 초과 시 **작업 전 사용자 사전 보고 필수**.

## 작업 순서 (의무)

1. **Baseline 확보 (필수)**: 수정 전에 **관련 테스트를 실제로 실행**한다 — 결과 명세서가 아니라 실행 로그가 필요
    - 멀티 모듈 프로젝트임: `./gradlew :test --tests "*패턴*"` (접두 `:test` 주의)
    - 환경 의존 실패 (Google Sheets 자격 증명 등) 가 있어도 OK — **변경 전후 동일한 실패 패턴이면 회귀 없음으로 판별 가능**
    - 테스트 부재 시: 사용자에게 보고하고 수동 검증 시나리오 합의 후 진행
    - 근거 (iteration-1 eval-1): with_skill 은 outputs 파일을 src 에 임시 swap → 실행 → 비교 → 즉시 복구 패턴으로 회귀 없음 확정. without_skill 은 baseline 실행 자체를 안 함.
2. **리스크 분석**: thread safety, 외부 제약(rate limit 등), 기존 TODO 주석 확인
3. **안전 확신 후 구현**. 불확실하면 먼저 보고
4. **재검증**: 같은 테스트 다시 실행. 수정 없이 통과 확인 (또는 1번과 동일한 실패 패턴)
5. **자가 보고** (자세한 형식은 아래 "자가 보고 형식" 섹션 참고)

## 의도 외 변경 발견 시 — 유지 + bug-fix 분기

리팩토링 도중 **사용자 요청 범위 외의 결함**을 발견하더라도 같은 PR 에서 손대지 말 것. 발견 → 박제 → 분리 보고.

### 차단 패턴 (이런 사유로 임의 수정 금지)

- ❌ "오타라 명백히 잘못된 거니 같이 고치자"
- ❌ "한 줄이라 비용도 작은데 같이 고치자"
- ❌ "외부 의존 없는 변경이라 안전해 보임"
- ❌ "어차피 동작에 영향 없는 코스메틱 변경"
- ❌ "리팩토링하다가 자연스럽게 같이 잡혀서 분리하기 어려움"

### 절차

1. 발견 즉시 **수정하지 않음**
2. notes 의 분리 보고 항목에 박제: 위치 + 현상 + 권장 후속 작업
3. 사용자에게 보고: "리팩토링 중 X 도 발견했는데 동작 변경이라 손대지 않았습니다. bug-fix 로 진행하시겠어요?"

근거 (iteration-1 eval-1): without_skill 은 `syncGraduationCertRule` 의 로그 메시지 오타 (`완료}` → `완료`) 를 임의 수정함. notes 에 "외부 로그 파싱 의존이 있다면 별도 합의 필요" 라고 적기는 했지만 이미 수정 후 보고 — **분기 결정권을 사용자에게서 빼앗음**. with_skill 은 "동작 보존 원칙으로 유지, bug-fix 분기 권장" 으로 정확히 처리.

**판단 기준**: 변경 후 행동이 1바이트라도 달라지면 동작 변경. 로그 메시지·예외 메시지·주석도 외부 시스템이 의존할 수 있음.

## 실패 처리 원칙

테스트 실패 시:

- ❌ **테스트를 수정해서 통과시키지 않는다**
- ✅ 실패 원인을 분석. 구현 문제면 롤백 후 재시도
- ✅ 기존 라이브러리/프레임워크의 숨겨진 동작(예: Spring 내부 상태)을 발견할 기회로 삼는다

근거 (Session 4-C, 2026-04-24): SSE 직렬화 리팩토링 시 초기 구현이 `ssePropagationTest` 실패 → Spring `SseEventBuilderImpl.build()` 의 trailing 개행 누적 동작 발견 → 전략 전환 → 통과. 이 경험이 "테스트 실패는 정보" 원칙의 근거.

## 자가 보고 형식 (필수)

작업 끝나면 다음 5개 섹션을 **모두 명시**. 해당 사항 없으면 "해당 없음" 으로 박제 — 누락은 작업 미완료로 간주.

1. **변경 파일 목록 + 라인 수** (CLAUDE.md '변경 범위 가이드' 한도 점검 결과 포함)
2. **변경 핵심 한 줄 요약** + 보존 동작 (어떤 시나리오·경로가 동일하게 작동하는지)
3. **Baseline + 재검증 결과** (실행 명령, 통과/실패 수, 변경 전후 비교)
4. **분리한 인접 문제** (있으면 항목별로 / 없으면 "해당 없음")
5. **적용 안 한 개선점** (있으면 / 없으면 "해당 없음")

근거 (iteration-1): without_skill 은 자가 보고 4·5 항목 누락 빈번. **빈 항목도 명시하면 "점검 안 함" 과 "점검 후 없음" 이 구별된다** — 이 구별이 사용자 신뢰의 핵심.
