# allcll 코드 분석 가이드

## 분석 프레임: 여러 축으로 나눠라

성능·품질 분석 시 최소한 다음 축을 **모두** 검토한다:

- **CPU 관점**: 직렬화·정렬·계산 반복
- **스레드 관점**: 블로킹, 점유 시간, pool 고갈 위험
- **I/O 관점**: DB 쿼리 수, 외부 API 호출, 네트워크 송신
- **N+1 관점**: 루프 내 쿼리, 컬렉션 순회 중 개별 조회
- **도메인 관점**: 수강신청 피크 특성, 실시간성 요구 등

한 축만 파면 놓치는 게 많다. (근거: Session 4-B — "학생 트래픽 비선형성" 프레임에만 집중해서 Admin 배치의 스레드 점유 문제를 놓침)

## 서브 에이전트 사용 시 주의

Explore 에이전트로 전체 구조 훑을 때 **파일을 카테고리로 분류**하는데,
이 분류가 곧 "관심도"를 결정하는 함정이 있다.

대응:

- 분류에서 제외된 섹션도 **주요 메서드 한 줄은** 확인
- "카테고리 상 관련 없어 보이는" 파일도 최소 1회 Read
- 의심 코드(`Thread.sleep`, `wait`, 긴 루프)는 섹션 무관하게 수집

## 의도 추정 필수

코드를 "고칠" 거라면, 고치기 전에 **"왜 현재 코드가 이렇게 작성됐는지"** 추정하고 "수정 전 확인 사항"을 명시한다.

예시 (Session 4):

- `SseEmitterStorage`의 TODO 주석 발견 → 팀이 이미 인지한 부분
- `wait(1000)`의 의도 = 외부 서버 rate limit → 제거하면 위험

## 수치 사용 원칙

- 벤치마크 없는 과장 수치 금지 ("90% 감소" 같은 것)
- 대신: 실제 규모 기반 계산 (예: "n=5,000일 때 수 ms 수준")
- 전제·가정도 "확인 필요" 표시 대상 — "동접 1만 명 전제"처럼 단정하지 말 것

## 결과 포맷

각 발견에 대해:

- 문제 코드 + **파일:라인 인용**
- 심각도 (🔴 높음 / 🟠 중간 / 🟡 낮음)
- 구체적 시나리오
- 해결 방향
- **의도 추정** (왜 현재 코드가 이런지)
- **수정 전 확인 사항**
